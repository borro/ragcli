# `internal/input`

## TL;DR

Пакет унифицирует вход: открыть файл по пути или материализовать `stdin` во временный файл и вернуть единый `Handle`/`Source`.

См. также [`doc/architecture.md`](../../doc/architecture.md).

## Зона ответственности

- открыть входной файл из `--file`;
- при отсутствии пути сохранить `stdin` во временный файл;
- вернуть `Source` с `Reader`, `Path` и `DisplayName`;
- удалить временный файл при cleanup.

## Ключевые entrypoints/types

- `Open(path, stdin)`
- `Handle`
- `Source`

## Входящие/исходящие зависимости

Входящие:

- `internal/app`

Исходящие:

- стандартная библиотека
- `internal/localize`

## Основной поток

1. Если путь задан, пакет просто открывает файл.
2. Если путь не задан, читается `stdin`.
3. Контент `stdin` записывается во временный файл.
4. Временный файл переоткрывается как reader.
5. `Handle.Close()` закрывает reader и удаляет временный файл.

## Инварианты и ошибки

- Все режимы получают path-aware источник даже при работе через `stdin`.
- Для `stdin` `DisplayName` равен `stdin`, а `Path` указывает на temp file.
- Ошибки открытия, записи, закрытия и повторного открытия локализуются.

## Что подтверждают тесты

- открытие файлов и обработка отсутствующего файла;
- материализация `stdin` и cleanup temp file;
- error paths при create/read/close/reopen temp file;
- безопасный `Close()` на nil/empty handle.

## Куда вносить изменения

- Любая новая политика работы со `stdin` или temp files должна жить здесь, а не в режимах.
- Если меняется форма `Source`, проверьте все mode-пакеты и `internal/app`.
